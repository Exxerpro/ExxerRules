Guiding Principles & Best Practices\n\nThis project is committed to excellence in software engineering, guided by the following principles:\n\n### 🧩 Design & Coding\n* **Core Principles:** Enforce SOLID, KISS, DRY, YAGNI, and the Law of Demeter rigorously.\n* **Methodology:** Apply SRP (Single Responsibility Principle), high cohesion, and low coupling.\n* **Code Quality:** Keep methods short, expressive, and side-effect-free.\n* **Error Handling:** Almost never throw exceptions. Prefer returning `Result<T>`.\n\n### 🧱 Architecture & Interfaces\n* **Modularity:** Design interfaces independent of technology, using minimal, focused abstractions.\n* **Interchangeability:** Ensure swappable modularity at compile-time or runtime.\n\n### ⚙️ Configuration & Usability\n* **Configuration:** Must be flexible, clear, and editable by users or admins.\n* **Reliability:** Incorporate built-in prediction, validation, and recovery mechanisms.\n* **UX/DevUX:** Deliver smooth, frictionless experiences and adhere to the principle of least surprise.\n* **Human-Centric:** Code should be designed for humans, not just compilers and programmers.\n\n> *\"Elegant execution is the mark of mastery.\"*\n\n---\n\n## 3. Development & Build Standards\n\n### 🏗️ Building\n* **Reproducibility:** Builds must be fully deterministic and repeatable across all environments.\n* **Automation:** No manual steps are allowed; builds must be reproducible via a CI/CD pipeline.\n* **Artifacts:** All artifacts must be versioned, traceable, and self-contained with metadata.\n* **Artifact Isolation:** All build outputs (tests, Stryker, NuGet packages) must live **outside the repository**. A `build.props` file should control output paths.\n\n### 🧪 Testing\n* **Mandatory Tests:** Tests are mandatory for all logic paths: unit, integration, and regression.\n* **Gatekeeping:** \"No test → No merge.\" All code must be test-covered with verifiable assertions.\n* **Qualities:** Tests must be fast, isolated, and idempotent. Failures should be actionable and localized.\n* **100% Coverage:** 100% code coverage is not the target. The real target is 100% feature completion and a 100% bug-free state within time and budget.\n* **Test Frameworks:** Use `XUnit.V3`, `Shouldly`, and `NSubstitute` for testing new projects. Avoid `FluentAssertions`, `Moq`, `MediaTr`, and `AutoMapper`.\n* **Pragmatism:** When testing, focus on behaviors and contracts, not internal implementation details.\n\n> *\"Tests aren't just validation—they are tomorrow's guarantees.\"*\n\n### 🛡️ Security\n* **Secure by Design:** Apply the principle of least privilege, fail-safe defaults, and defense-in-depth.\n* **Input Validation:** All inputs must be validated, sanitized, and verified without exception.\n* **Secrets Management:** Secrets must be managed via vaults and never hardcoded or stored in code/config files.\n\n### 📡 Observability\n* **Self-Explanation:** Systems must be self-explaining, with required metrics, logs, traces, and health checks.\n* **Telemetry:** All components must emit structured, timestamped, and correlated telemetry.\n* **Alerting:** Alerts are mandatory and must be meaningful, real-time, and testable.\n\n---\n\n## 4. Code Style & Conventions\n\n* **No Regions:** Prefer sub-classes instead of regions for organizing code.\n* **Meaningful Names:** Use descriptive, clear, and consistent names for all variables, functions, and classes. Maintain XML comments for tooling and IntelliSense.\n* **Avoid Globals & Hardcoding:** Do not use global variables. Always define constants for \"magic numbers\" and \"magic strings\".\n* **Dependencies:** Be mindful of external dependencies and their versions.
