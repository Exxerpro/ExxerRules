# UseXUnitV3 Analyzer - False-Positive Mitigation Spec

**Analyzer ID**: `EXXER101`  
**Source**: `src/code/IndFusion.Analyzer/Testing/UseXUnitV3Analyzer.cs`  
**Prepared by**: Codex agent (2025-10-07)

## 0. Selection Rationale

- `Directory.Build.runsettings:6` disables EXXER101 via `<NoWarn>EXXER101</NoWarn>` for the entire `Test Project`, indicating pervasive noise that blocks adoption.  
- Code generated by the Microsoft Testing Platform (located under `Test Project/Src/Tests/**/obj/Debug/net10.0/`) still carries legacy MSTest hooks (`Microsoft.Testing.Platform.*`), and the analyzer treats those assembly-level attributes as MSTest usage even though they are auto-generated shims for xUnit v3.  
- Several integration harnesses intentionally reference MSTest/NUnit APIs to verify migration tooling (see `src/test/IndFusion.Analyzer.Tests/TestCases/TestingStandardsTests.cs`), but shipping code should not be warned. Without an opt-out, developers suppress EXXER101 globally.

## 1. Specification

- **Intent**  
  Ensure all newly authored tests target xUnit v3 by flagging references to alternative unit-test frameworks (MSTest/NUnit) via attributes or namespace imports.

- **Scope**  
  Registers `SyntaxKind.Attribute` and `SyntaxKind.UsingDirective`. It raises warnings when it sees attributes named `Test`, `TestMethod`, `TestCase`, etc., or when using directives reference `Microsoft.VisualStudio.TestTools.UnitTesting`, `NUnit.Framework`, or `NUnit`.

## 2. Validation Plan

1. Create a new `UseXUnitV3AnalyzerFalsePositiveTests` class with the mitigation scenarios below plus the existing positive cases (MSTest attribute and namespace should continue to trigger).  
2. Update `TestingStandardsTests` to cover the new heuristics (auto-generated assemblies, compatibility fixture projects, explicit allowlists).  
3. Run `dotnet test src/test/IndFusion.Analyzer.Tests/IndFusion.Analyzer.Tests.csproj -c Release` to confirm all analyzer tests pass after the mitigations.  
4. Remove `<NoWarn>EXXER101</NoWarn>` from `Directory.Build.runsettings` and rebuild IndTrace solution to ensure diagnostics are limited to true MSTest/NUnit usage.

## 3. Enhancement Opportunities (≥10 Items)

Each item records an observed or reported false positive, the proposed mitigation, and a regression test snippet (xUnit + Shouldly) that currently fails and will pass after the fix.

### 1. Skip Auto-Generated Microsoft Testing Platform Files

- **Problem**: Files like `Test Project/Src/Tests/Presentation/MetricsTests/obj/Debug/net10.0/SelfRegisteredExtensions.cs` contain `[global::Microsoft.Testing.Platform.MSBuild.TestingPlatformBuilderHook]` but are `<auto-generated>`. EXXER101 treats them as MSTest usage.  
- **Mitigation**: Ignore attributes/using directives in files that include an `<auto-generated>` comment or reside under `obj/`/`bin/`.  
- **Test**:

```csharp
[Fact]
public void Should_Not_Report_For_AutoGenerated_TestingPlatform_Shims()
{
    const string testCode = @"
// <auto-generated>
using Microsoft.Testing.Platform;

[assembly: Microsoft.Testing.Platform.MSBuild.TestingPlatformBuilderHook]
namespace Generated { }";

    AnalyzerTestHelper.RunAnalyzer(testCode, new UseXUnitV3Analyzer(),
        filePath: @"C:\repo\Tests\obj\Debug\net10.0\SelfRegisteredExtensions.g.cs")
        .ShouldBeEmpty();
}
```

### 2. Allow Microsoft Testing Platform Builder Hooks in Source

- **Problem**: Even when developers check in hook files (for IDE convenience), the analyzer flags `Microsoft.Testing.Platform.*` attributes.  
- **Mitigation**: Treat `Microsoft.Testing.Platform.*` as an approved namespace since it composes xUnit v3.  
- **Test**:

```csharp
[Fact]
public void Should_Not_Report_For_TestingPlatform_BuilderHooks()
{
    const string testCode = @"
using Microsoft.Testing.Platform.MSBuild;

[TestingPlatformBuilderHook]
public static class Hook { }";

    AnalyzerTestHelper.RunAnalyzer(testCode, new UseXUnitV3Analyzer())
        .ShouldBeEmpty();
}
```

### 3. Ignore Compatibility Fixtures Referencing MSTest for Migration Proofs

- **Problem**: Integration harnesses under `src/test/IndFusion.Analyzer.Tests` intentionally import MSTest (`using Microsoft.VisualStudio.TestTools.UnitTesting;`) to validate migration code fixes. They should remain as positive controls within analyzer tests, not leak into solution diagnostics.  
- **Mitigation**: Allow opt-in via `AnalyzerConfigOptions` (e.g., `dotnet_diagnostic.EXXER101.ignore_projects = IndFusion.Analyzer.Tests`).  
- **Test**:

```csharp
[Fact]
public void Should_Respect_AnalyzerConfig_Ignore_List()
{
    const string testCode = @"
using Microsoft.VisualStudio.TestTools.UnitTesting;

[TestClass]
public class LegacyTests
{
    [TestMethod] public void Should_Work() { }
}";

    var options = AnalyzerOptionsFactory.Create(
        ("dotnet_diagnostic.EXXER101.ignore_projects", "IndFusion.Analyzer.Tests"));

    AnalyzerTestHelper.RunAnalyzer(testCode, new UseXUnitV3Analyzer(),
        analyzerOptions: options, projectName: "IndFusion.Analyzer.Tests")
        .ShouldBeEmpty();
}
```

### 4. Permit Local Compatibility Assemblies

- **Problem**: Some teams ship interop layers referencing MSTest attributes but targeting xUnit (e.g., wrappers providing `[TestMethod]` bridging). Analyzer flags those wrappers.  
- **Mitigation**: If an attribute resolves to a symbol defined within the current solution (custom shim), skip reporting.  
- **Test**:

```csharp
[Fact]
public void Should_Not_Report_For_Local_Wrapper_Attributes()
{
    const string testCode = @"
namespace Compat
{
    public class TestMethodAttribute : System.Attribute { }
}

[Compat.TestMethod]
public class Sample { }";

    AnalyzerTestHelper.RunAnalyzer(testCode, new UseXUnitV3Analyzer())
        .ShouldBeEmpty();
}
```

### 5. Allow `Microsoft.Testing.Platform` Namespaces in `using` Directives

- **Problem**: Auto-generated entry points (`XunitAutoGeneratedEntryPoint.cs`) include `using Microsoft.Testing.Platform.*`, which the analyzer counts as forbidden because of the substring "Testing".  
- **Mitigation**: Expand the namespace allowlist to include `Microsoft.Testing.Platform` roots.  
- **Test**:

```csharp
[Fact]
public void Should_Not_Report_For_TestingPlatform_Using()
{
    const string testCode = @"
using Microsoft.Testing.Platform.Builder;

public static class Boot { }";

    AnalyzerTestHelper.RunAnalyzer(testCode, new UseXUnitV3Analyzer())
        .ShouldBeEmpty();
}
```

### 6. Distinguish Between Sample Code Strings and Attributes

- **Problem**: Documentation samples embed `[TestMethod]` inside verbatim strings or comments; the current analyzer (working off `attribute.Name.ToString()`) catches them when samples are placed inside `#if DEBUG` blocks as part of the compilation.  
- **Mitigation**: Ignore attributes within inactive `#if` regions or inside string literals.  
- **Test**:

```csharp
[Fact]
public void Should_Not_Report_For_Sample_Code_In_Strings()
{
    const string testCode = @"
public sealed class Docs
{
    public const string Sample = \"[TestMethod] public void Legacy() { }\";
}";

    AnalyzerTestHelper.RunAnalyzer(testCode, new UseXUnitV3Analyzer())
        .ShouldBeEmpty();
}
```

### 7. Provide Configuration to Allow Specific Legacy Assemblies

- **Problem**: IndTrace ships a `Compatibility.LegacyMSTest` assembly used only to host legacy tests; currently EXXER101 fires on every file.  
- **Mitigation**: Support per-assembly overrides via `AssemblyAttributes` (e.g., `[assembly: SuppressTestFrameworkConversion("MSTestLegacy")]`).  
- **Test**:

```csharp
[Fact]
public void Should_Honor_SuppressTestFrameworkConversion_Attribute()
{
    const string testCode = @"
[assembly: SuppressTestFrameworkConversion(""MSTestLegacy"")]
using Microsoft.VisualStudio.TestTools.UnitTesting;

[TestClass]
public class Legacy { [TestMethod] public void Old() { } }";

    AnalyzerTestHelper.RunAnalyzer(testCode, new UseXUnitV3Analyzer())
        .ShouldBeEmpty();
}
```

### 8. Ignore Analyzer Test Assets

- **Problem**: Files under `src/test/IndFusion.Analyzer.Tests/TestCases` intentionally reference MSTest/NUnit to assert diagnostics. The analyzer should not re-run against its own test assets when bundled in the solution.  
- **Mitigation**: Skip analysis when `ProjectName` ends with `.Analyzer.Tests` (configurable) or when `EditorBrowsableAttribute` indicates test asset.  
- **Test**:

```csharp

using NUnit.Framework;

[TestFixture]
public class AnalyzerFixture { }";

    AnalyzerTestHelper.RunAnalyzer(testCode, new UseXUnitV3Analyzer(),
        projectName: "IndFusion.Analyzer.Tests")

```

### 9. Avoid Double-Reporting on Transitive `using` Statements

- **Problem**: `global using Microsoft.VisualStudio.TestTools.UnitTesting;` inside a shared `Imports.cs` produces duplicate diagnostics in every file that merges the global using.  
- **Mitigation**: Track reported namespaces per syntax tree to ensure each offending `using` is flagged once.  
- **Test**:

```csharp
[Fact]
public void Should_Report_GlobalUsing_Only_Once()
{
    const string testCode = @"
global using Microsoft.VisualStudio.TestTools.UnitTesting;

[TestClass]
public class Legacy { }";

    var diagnostics = AnalyzerTestHelper.RunAnalyzer(testCode, new UseXUnitV3Analyzer());
    diagnostics.Count(d => d.Id == DiagnosticIds.UseXUnitV3).ShouldBe(1);
}
```

### 10. Provide Severity Downgrade/Info Mode for Transitional Projects

- **Problem**: While migrating, teams want informational reminders rather than blocking warnings. Lack of configuration leads to blanket suppression (as seen in `Directory.Build.runsettings`).  
- **Mitigation**: Respect `dotnet_diagnostic.EXXER101.severity` from `.editorconfig` to allow `suggestion` or `silent` severity without disabling the rule.  
- **Test**:

```csharp
[Fact]
public void Should_Respect_Severity_From_EditorConfig()
{
    const string testCode = @"
using Microsoft.VisualStudio.TestTools.UnitTesting;

[TestClass]
public class Legacy { }";

    var options = AnalyzerOptionsFactory.Create(
        ("dotnet_diagnostic.EXXER101.severity", "suggestion"));

    var diagnostics = AnalyzerTestHelper.RunAnalyzer(
        testCode, new UseXUnitV3Analyzer(), analyzerOptions: options);

    diagnostics.Single().Severity.ShouldBe(DiagnosticSeverity.Info);
}
```

## 4. Test-Driven Fix Strategy

- Add `UseXUnitV3AnalyzerFalsePositiveTests` with the ten scenarios above plus existing positive-control tests (ensuring `[TestMethod]` still triggers when no suppression applies).  
- Extend `UseXUnitV3Analyzer` to:
  - Skip auto-generated files and `obj/`/`bin/` paths.  
  - Allow `Microsoft.Testing.Platform.*` namespaces/attributes.  
  - Resolve attribute symbols to distinguish custom shims from MSTest/NUnit assemblies.  
  - Respect analyzer configuration (`ignore_projects`, `severity`).  
  - Deduplicate diagnostics for `global using` statements.  
- Wire configuration helpers into `AnalyzerTestHelper` to support project-level metadata used in the new tests.  
- Run analyzer tests and remove `<NoWarn>EXXER101</NoWarn>`; rebuild IndTrace solution to confirm diagnostics are limited to genuine MSTest/NUnit usage or are downgraded per configuration.
