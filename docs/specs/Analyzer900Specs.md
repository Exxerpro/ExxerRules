# ProjectFormatting Analyzer - False-Positive Mitigation Spec

**Analyzer ID**: `EXXER900`  
**Source**: `src/code/IndFusion.Analyzer/CodeFormatting/ProjectFormattingAnalyzer.cs` (paired with `CodeFormattingAnalyzer.cs`)  
**Prepared by**: Codex agent (2025-10-08)

## 0. Selection Rationale

- `docs/specs` has no coverage for EXXER900 even though `AnalyzerReleases.Unshipped.md:20` ships it as an always-on hidden diagnostic.  
- `RulesAnalysis.md:169-175` already flags EXXER900/901 as high-noise analyzers whose heuristics routinely contradict `.editorconfig`.  
- The hidden trigger currently surfaces across every syntax tree, including generated assets like `src/ExxerRules/ExxerAI.Rules.Analyzer/obj/Debug/netstandard2.0/.NETStandard,Version=v2.0.AssemblyAttributes.cs`, and the coupled formatting analyzer (`CodeFormattingAnalyzer`) reports diagnostics on perfectly formatted IndTrace source (`Test Project/Src/Code/Core/Application/BarCodes/Commands/Create/CreateBarCodeCommandHandler.cs`) because its string-based spacing checks misfire. These issues block adoption—teams disable EXXER900/901 via `.editorconfig` instead of using the intended formatting workflows.

## 1. Specification

- **Intent**  
  Provide IDE/light-bulb entry points for running `dotnet format` at project scope (EXXER900) and raise info diagnostics when formatting issues are detected (EXXER901) so developers can trigger formatting actions quickly.

- **Scope**  
  - `ProjectFormattingAnalyzer` registers `SyntaxTreeAction` against every C# tree and emits a hidden diagnostic at the first node (or span 0) to advertise project-level formatting commands. It performs no analysis and ignores file type, generated-code markers, or configuration overrides.  
  - `CodeFormattingAnalyzer` inspects class, method, property, and variable declarations with purely textual heuristics. It looks for substrings like `" ="`, `"= "`, multi-line parameter lists, and adjacency of members to approximate brace and spacing violations.

## 3. Enhancement Opportunities (≥10 Items)

Each item lists the current false-positive, a mitigation idea, and an xUnit + Shouldly snippet that fails today and should pass after the fix.

### 1. Generated Files Trigger Hidden Diagnostics

- **Problem**: `ProjectFormattingAnalyzer` emits EXXER900 on generated assets such as `src/ExxerRules/ExxerAI.Rules.Analyzer/obj/Debug/netstandard2.0/.NETStandard,Version=v2.0.AssemblyAttributes.cs#L1` (`// <autogenerated />`). These files should never surface formatting actions.  
- **Mitigation**: Use `SyntaxTreeAnalysisContext.GetText().StartsWith("// <auto-generated")` (and file-path filters containing `\obj\`/`\bin\`) to bail out before reporting.  
- **Test**:

```csharp
[Fact]
public void Should_Not_Report_For_AutoGenerated_Files()
{
    const string code = @"// <autogenerated />
namespace Samples;
public class Generated { }";

    AnalyzerTestHelper.RunAnalyzer(
        code,
        new ProjectFormattingAnalyzer(),
        includeHidden: true,
        filePath: @"C:\repo\MyProject\obj\Debug\net8.0\Generated.g.cs")
    .ShouldBeEmpty();
}
```

### 2. Empty/Whitespace Files Still Raise EXXER900

- **Problem**: An empty file or one containing only comments still satisfies `root.HasLeadingTrivia` and surfaces EXXER900, producing a useless light bulb.  
- **Mitigation**: Require at least one non-trivia token before reporting.  
- **Test**:

```csharp
[Fact]
public void Should_Not_Report_For_Empty_Files()
{
    const string code = @"// intentional placeholder";

    AnalyzerTestHelper.RunAnalyzer(
        code,
        new ProjectFormattingAnalyzer(),
        includeHidden: true,
        filePath: @"C:\repo\Project\Empty.cs")
    .ShouldBeEmpty();
}
```

### 3. Hidden Diagnostic Anchors to Irrelevant Location

- **Problem**: EXXER900 anchors to the first child node (often the `namespace` declaration). In files like `Test Project/Src/Code/Core/Application/Registers/Services/RegisterInformationService.cs:10`, the code action lights up the namespace even though formatting actions relate to local statements.  
- **Mitigation**: Report the diagnostic on a zero-length span at the start of the tree (`Location.Create(tree, new TextSpan(0, 0))`) to avoid confusing highlights, or attach to `CompilationUnit`.  
- **Test**:

```csharp
[Fact]
public void Should_ReportAtCompilationStart_When_ProjectFormatting()
{
    const string code = @"namespace Sample;
public class C { }";

    var diagnostic = AnalyzerTestHelper.RunAnalyzer(
        code,
        new ProjectFormattingAnalyzer(),
        includeHidden: true,
        filePath: @"C:\repo\Project\Sample.cs").Single();

    diagnostic.Location.SourceSpan.Start.ShouldBe(0);
}
```

### 4. LINQ Assignments Flagged as “Inconsistent Variable Formatting”

- **Problem**: `HasInconsistentVariableFormatting` inspects raw strings and flags every initializer containing `" ="` or `"= "`. Legitimate assignments like `var availableRegisters = availableRecords.Select(...)` (`Test Project/Src/Code/Core/Application/Registers/Services/RegisterInformationService.cs:24`) fire EXXER901 despite perfect spacing.  
- **Mitigation**: Tokenize the declaration and ensure both sides of the equals token include whitespace trivia, rather than substring matching.  
- **Test**:

```csharp
[Fact]
public void Should_Not_Report_For_Linq_Assignments()
{
    const string code = @"
using System.Linq;
namespace Samples;
public static class Target
{
    public static void Build()
    {
        var availableRegisters = Enumerable.Range(1, 3)
            .Select(id => $""PLC{id:D3}"");
    }
}";

    AnalyzerTestHelper.RunAnalyzer(code, new CodeFormattingAnalyzer())
                     .ShouldBeEmpty();
}
```

### 5. Awaited Repository Calls Misreported

- **Problem**: Assignments like `var registerResult = await registerRepository.ListAsync(...)` (`RegisterInformationService.cs:103`) are flagged because the analyzer still finds `" ="` substrings even though spacing is correct.  
- **Mitigation**: Recognize `await` expressions and confirm the equals token has leading/trailing whitespace trivia before reporting.  
- **Test**:

```csharp
[Fact]

    }

    AnalyzerTestHelper.RunAnalyzer(code, new CodeFormattingAnalyzer())
                     .ShouldBeEmpty();
}
```

- **Problem**: `var mockData = Enumerable.Range(1, 5).ToDictionary(...)` (`RegisterInformationService.cs:187`) also emits EXXER901 due to the same substring heuristic.  
- **Mitigation**: After token inspection, allow expressions that already contain spaces around `=` and are part of `InvocationExpressionSyntax`.  
- **Test**:

- **Problem**: In `CreateBarCodeCommandHandler.cs:95-112`, assignments such as `var spec = new Specification<Machine>(...)` are flagged, flooding EXXER901 through fluent predicate builders.  
- **Mitigation**: Treat generic `new` expressions with properly spaced tokens as compliant; only flag when the equals token lacks either leading or trailing whitespace.

### 10. `var result = await Result.Success(...)` Chains Flagged

- **Problem**: Fluent result pipelines in `CreateBarCodeCommandHandler.cs:63` start with `var result = await Result.Success(...)`, flagged as formatting issues despite perfect whitespace.  
- **Mitigation**: Recognize the pattern `var identifier = await` with proper space tokens and skip diagnostics.  
- **Test**:


## 4. Test-Driven Fix Strategy

- Expand the code formatting analyzers to rely on `SyntaxToken` and trivia inspection (tokens before/after `SyntaxKind.EqualsToken`, `SyntaxKind.OpenBraceToken`, etc.) rather than raw string searches.  
- Whitelist generated files and `obj`/`bin` directories in `ProjectFormattingAnalyzer`, and ensure diagnostics attach to the compilation start span.  
- Extend `AnalyzerTestHelper` with overloads for `includeHidden` and `filePath` (already needed above).  
- Run the analyzer/unit test suite (`dotnet test ...`) and rebuild IndTrace sample projects to confirm EXXER900/901 produce only actionable diagnostics.
